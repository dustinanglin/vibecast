01 · OVERVIEW

What's new in this round.

The current app is functional — a list of subscribed shows, a player, a mini-player. This handoff adds a layer of intent on top: how someone listens at different times of day, in different moods, on different occasions.

The big idea: vibes

A vibe is a manually-ordered queue of shows for a moment in time — Morning routine, Around town, Wind down, On a plane. Tap "Start the vibe" and it plays the latest of show #1, then #2, then #3, in your order.

Vibes give the library structure that subscriptions alone can't. They turn the question "what do I want to play?" into "what kind of moment am I in?".

The model in one sentence

A vibe holds two kinds of items: show items (always plays the latest unplayed episode) and pinned episodes (a specific episode that expires from the queue when played).

This is what makes "On a plane" work: subscribe to 3 podcasts, pin 4 specific older episodes for the flight, and the queue cleans itself up when you land.

DESIGN STANCE

Editorial, paper-warm, dark text. Color belongs to vibes — each one is a single hue from a shared oklch family. The library stays calm so the colored vibe context can carry meaning when it appears.

02 · BRAND

Wordmark, icon, splash.

One brand thread carries through three places: the wordmark, the app icon, and the splash screen. All three share the same letterform (Fraunces serif v) and the same accent dot — the dot is the live, vibe-colored element that ties the brand to the active mood.

Wordmark · primary

Fraunces 500, letter-spacing -0.035em, accent dot trailing the final t. The dot inherits the color of the most-recently-played vibe.

App icon · v + period

Splash · animated side bar

On first launch the splash plays a short cycle through the five vibes — a vertical color bar pinned to the left edge, the dot in the wordmark, and a small label chip all crossfade together. 04 · DATA MODEL

What lives where.

The current Swift model already has shows and episodes. This adds two concepts: vibes (named, ordered, colored containers) and queue items (the polymorphic thing inside a vibe).

Vibe

struct Vibe {
  let id: UUID
  var name: String         // "On a plane"
  var hue: Double          // 0–360, oklch H
  var queue: [QueueItem]   // user's manual order
  var isTemporary: Bool    // future — purges itself
}

A vibe owns its queue order. No separate "ordering" table needed.

QueueItem (sum type)

enum QueueItem {
  case show(Show.ID)                    // "latest unplayed of X"
  case episode(Episode.ID,
               pinnedAt: Date)          // specific episode, expires on play
}

The two kinds reorder freely together. Pinned items leave the queue when they finish.

Constraints & cardinality

Show in N vibes
0–2. Soft cap. Beyond that, the multi-vibe gradient gets muddy and the mental model frays.
Episode pins
Any vibe, any number. Pins are temporary by design — caps not needed. A user can pin 6 Blank Check episodes to "On a plane" without it polluting "Around town".
Pin → vibe
A pin does not require the show to be a member of that vibe. This is what lets "On a plane" pull from anywhere.
Played pinned ep
Removed from queue. Stays in the show's "played" history (existing data).
"All vibes" view
Master order across all subscribed shows. Independent from per-vibe orders.

Right-side control reflects the action

  • ⏸Now — vibe-color filled circle, white pause glyph. Tap pauses.
  • ▶Paused — same circle, play glyph. Tap resumes.
  • ▶Queued / Up Next — paper-colored ring, ink glyph. Tap skips to this row.
  • ↻Played — transparent, muted glyph. Tap replays.

Left-side position slot is multi-purpose

  • NPosition number for queued shows (zero-padded mono).
  • ≡Animated 3-bar VU when actively playing.
  • 📌Pin glyph (vibe-colored) for pinned episodes.
  • ⋮⋮Drag handle in reorder mode.

Started rows — partially played, not active

A separate state from now and played. The user paused and may come back. The treatment downplays it relative to unplayed neighbors so the queue's CTA stays on the next thing to play.

Title
Fraunces 300 italic at 78% ink (vs 500 upright at full ink for unplayed). The italic is the carrier of “in progress, not done.”
Eyebrow
Inline progress ring (11px, vibe color) + {N}M LEFT in vibe ink, then · {Show name} in muted ink. “Left” reads more queue-useful than “in.”
Footnote
PAUSED AT {N}M · {TOTAL} MIN TOTAL in muted mono beneath the title. Unplayed rows show the simpler {TOTAL} MIN in the same slot — same treatment, less data.
Position number
Stays in the left slot (no progress ring there). Dimmed to 30% so unplayed rows still win attention. The number reinforces “you decided this order.”
No left sliver
Started rows do not get the left-edge color sliver. The sliver is reserved as an attractor for unplayed rows.

Left-edge sliver — the unplayed attractor

A 3px vibe-colored bar inset from the row's left edge. Lives only on unplayed rows. Same component, two visual personalities depending on context:

All Vibes view
Sliver uses each show's primary vibe (first item in its tags). The result is a chromatic column down the left edge — the home view feels like a paint chip lineup without shouting.
Specific vibe view
Every row shares that vibe, so all slivers are uniform. Reads as a clean section marker.
Multi-vibe shows
Use the primary only. A split sliver is too small to read as intentional at row scale; the chip-tint background already carries the second vibe.
No vibes
Untagged shows fall back to ink. Same component, every state.

Now-playing row — the active card

When a row is the active item (status = now or paused), it lifts off the list with a vibe-color frame, a glow, and a live progress bar across the top edge. Same row template — the framing does the work.

Border
2px solid {vibe.color} (vs 1px solid hairline for inactive). Doubles up the edge weight so it reads as the focused row even at a glance.
Halo
box-shadow: 0 8px 24px {vibe.color}33 — a soft vibe-tinted lift. Inactive rows have no shadow.
Top progress bar
3px sliver pinned to the row's top edge. Track is rgba(0,0,0,0.08); fill is {vibe.color} at width: {progressPct}%. Lives flush with the row's rounded corners (overflow: hidden on the card).
Left slot
Position number is replaced with a 20×20 vibe-color circle. Now shows a 3-bar VU animation in white inside it; paused shows two white bars. The circle stays the same size — only its glyph changes.
Right control
Filled vibe-color circle (38×38) with white glyph — pause when playing, play when paused. The only filled circle in the row stack; everything else is outlined.
Eyebrow
{TOTAL} MIN · {N}M IN — the in half is rendered in vibe ink so the live progress reads twice (top bar visually, eyebrow numerically).
Implementation
See PlaybackRow in vibes-playback.jsx. The isActive branch covers both now and paused — only the glyph and right-button icon differ.

07 · INTERACTIONS

How it moves.

Start a vibe

  • 1Tap a vibe pill → page collapses to that vibe's queue, header band picks up the vibe's chip color, "Start the vibe" CTA appears in vibe color. The vibe is queued but not playing yet.
  • 2Tap "Start the vibe" → first item begins. Header switches to live "VIBING" with pulsing dot. Mini player appears. CTA hides.
  • 3Episode finishes → auto-advance to next item in order. Played row dims, next row's eyebrow flips from "UP NEXT" to active.
  • 4All items finished → "Vibe complete" screen. Restart, or pick another vibe.

Pin an episode

  • 1From a podcast detail page, tap the pin icon on any episode (or the hero "Pin to vibe" button to pin the latest).
  • 2Sheet slides up. Pick a vibe — selected state shows in that vibe's color. Tap the filled CTA.
  • 3Sheet dismisses. Pin icon on the source row turns vibe-colored. The vibe screen now contains this episode in queue order (default: appended to end; user can drag to reorder).
  • 4When played — pinned item leaves the queue silently. Show items the user subscribed to remain.

Skip / replay

  • ·Tap a queued row → skip to that row. Plays from the start.
  • ·Tap a played row's ↻ button → restart from the beginning. Row un-dims.
  • ·Tap the row anywhere outside the right-side button → tappable region is the same as the right-side play button. Whole row is the affordance; the button is the explicit one.

Reorder

  • ·Tap "Edit" in the queue header → enters reorder mode. Position glyphs swap to drag handles. Right-side controls collapse to keep targets clear.
  • ·Drag rows freely. Pinned and show items mix in any order.
  • ·Tap "Done" → exits, persists order on the vibe.
08 · RULES & EDGE CASES

Decided.

  • ·Multi-vibe shows get a soft gradient from vibe-A chip to vibe-B chip on their row. Cap at 2 vibes per show — beyond that, gradient and mental model both break down.
  • ·"Start the vibe" plays in order, skipping already-played episodes. Auto-prune on; the queue is always fresh.
  • ·Pin = latest of show? Dedupe — show only the pinned version (user explicitly chose its position). When played, the show item takes over with the next-newest unplayed episode.
  • ·Partial play of a pinned episode stays in queue with playback position preserved. Only completion expires it.
  • ·Manual unpin behaves identically to playing — the item leaves the queue. Long-press a pinned row for inline unpin; sheet also offers it.
  • ·Show with zero vibes still appears in "All" with no tint. Empty cards stand out as "ungrouped" on purpose.
  • ·Color collisions avoided by curating the vibe color picker to ~10–12 hues from the family. No free color picker.
09 · ITERATIVE ADOPTION

How to land it without breaking the app.

The current app is a flat list + player. The new model is additive — vibes can ship in stages without disrupting the existing flow.

PHASE 1

Visual refresh

Adopt the editorial palette + type pairing. Refresh library, podcast detail, Now Playing. No data model changes. The app already works; it just looks intentional.

PHASE 2

Vibes

Add Vibe + show-only QueueItem. Build vibe filter, vibe screen, vibe-aware Now Playing & mini player. Manual order. "Start the vibe."

PHASE 3

Pinning

Add episode-kind QueueItem. Pin sheet, pin icons on episode rows, expiration on play. Reorder mixed queues. The "On a plane" use case lights up.

What's already there vs what's net-new

Already in app
Subscriptions, episode list, player + mini player, played history, mark-as-played
Visual changes only
Library list, podcast detail, Now Playing, mini player — same data, new look
Net-new model
Vibe entity, QueueItem sum type, per-vibe ordering, pinned-at timestamp
Net-new screens
Vibe filter pills, vibe screen, pin sheet, vibe complete screen, reorder mode
Net-new behaviors
Auto-advance through queue, expire-on-play, dedupe pin↔latest, soft 0-2 vibe cap
10 · STILL OPEN

To decide on the way to ship.

  • ·Truly temporary vibes — a vibe that auto-deletes after every item is played or expires. "On a plane" might want this. Maybe a property on the Vibe itself.
  • ·Smart suggestions — at 7am, surface "Morning routine" first. Time-of-day, day-of-week, location. Could be a separate "Suggested" rail above pinned vibes; needs a privacy stance.
  • ·Cross-device sync — vibe order, pins, played-state. Existing iCloud sync on subscriptions probably extends, but pin timestamps need the same treatment.
  • ·Accessibility — vibe color shouldn't be the only signal. Verify the eyebrow text + position glyph + pill carry the meaning when color is invisible.
  • ·Wordmark in app chrome — should the live wordmark in the library header tint its dot to the currently-vibing vibe? Currently spec'd as static / most-recent.
  • ·Vibe complete tagline — the splash uses per-vibe taglines; consider reusing the same copy on the “Vibe complete” screen.
